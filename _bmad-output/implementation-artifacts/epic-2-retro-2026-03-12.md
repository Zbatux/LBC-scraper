## Challenges

### Out-of-Scope Fix in Code Review (Story 2.2)

**Issue:** Story 2.2 code review identified that `parse_date_publication` in `parsers.py` needed to handle Unix timestamp integers. The fix was applied directly to `parsers.py` — a file not in the story's scope.
**Impact:** Introduced 1 broken test (`test_coerces_non_string_to_string`) that persisted into Story 2.3 as a "pre-existing failure."
**Root cause:** Code review fixes were applied in the same commit as story changes, without separation.
**Learning:** Out-of-scope fixes should be handled as separate quick-specs to avoid cross-contamination.

### Test Failure Accepted as "Pre-existing" (Story 2.3)

**Issue:** 1 test was failing (introduced by Story 2.2's parsers.py fix) and was accepted as "pre-existing, out of scope" during Story 2.3.
**Impact:** Broken test persisted for 1 full story cycle without correction.
**Learning:** Broken tests should be fixed immediately, not deferred.

### Untracked Development (Post-Epic)

**Issue:** The "Compare similar lands" feature (commit 85a6759) was developed outside the sprint plan after Epic 2 completion. It merged the "Lien" and "Historique" columns into a single "Actions" column.
**Impact:** The history button became invisible to the user (only shows when `history_count > 0`, which is always 0 since no second scrape has been run). The user reported losing access to the history view.
**Learning:** All development should go through sprint tracking (`/bmad-bmm-quick-spec` + `/bmad-bmm-quick-dev`) to maintain visibility and avoid UI regressions.

---

## Key Insights

1. **Retro action items work when applied systematically.** 100% follow-through on Epic 1 actions produced measurable quality improvements in Epic 2.
2. **Out-of-scope fixes during code review create side effects.** Separate them into quick-specs to maintain story isolation.
3. **Untracked development causes silent regressions.** The "Comparer" feature broke the history button visibility without anyone noticing until the user reported it.
